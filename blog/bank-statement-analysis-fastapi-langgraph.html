Building an NBFC Bank Statement Analysis Platform with FastAPI, LangGraph & Next.js

By Nitesh Kumar Singh · NKSCoder · AI Developer & Senior Software Engineer · June 18, 2026

This is a full-stack fintech platform I built as NKSCoder — a bank statement analysis system for NBFCs, loan agents, banks, and fintech lenders. Upload PDF or Excel statements from any major Indian bank, extract transactions automatically, run AI-powered risk analysis, chat with the data in simple Hindi-English, and download a professional credit assessment PDF report.

Related production work: Ollama + GPU case study at Airtel Payments Bank

Who This Is For

Loan agents and NBFC underwriters spend hours manually reading bank statements — counting salary credits, finding EMI debits, spotting cheque bounces, and estimating whether a customer can afford another loan. This platform automates that workflow end-to-end:

  • Upload — single or bulk PDF / Excel / CSV statements
  • Parse — auto-detect bank, find header rows, extract transactions
  • Categorize — Salary, EMI, UPI, Shopping, Bounce, and more
  • Analyze — LangGraph AI pipeline for fraud, cashflow, income, and EMI burden
  • Chat — ask questions in English or Hinglish across one or multiple statements
  • Report — download NBFC-standard credit assessment PDF

The Problem: Indian Bank Statements Are Messy

Unlike clean API feeds, real bank statements arrive as PDFs and Excel exports with wildly different layouts. HDFC puts headers on row 3; SBI uses “Withdrawal Amt.” instead of “Debit”; Axis exports have title rows before column names. A parser hardcoded to row 0 and fixed column names breaks on the next bank export.

Key challenges I solved in this build:

  • No fixed header row — scan first 20 rows, score each for date/narration/debit/credit columns
  • Synonym matching — “Narration”, “Particulars”, “Transaction Details” all map to description
  • Multi-bank support — HDFC, SBI, ICICI, Axis, Kotak, PNB, BOB, Canara, Union, IDBI, Yes, IndusInd, Bandhan
  • Amount formats — handle 1,234.56 Dr, parentheses negatives, and empty cells
  • PDF tables + text fallback — pdfplumber table extraction with line-by-line fallback

Tech Stack

  • Frontend — Next.js 15, React 19, Tailwind CSS
  • Backend — FastAPI, SQLAlchemy, Pydantic, Python 3.12
  • Database — PostgreSQL 16
  • PDF parsing — pdfplumber
  • Excel parsing — openpyxl + flexible header detection
  • AI orchestration — LangGraph (multi-agent pipeline)
  • LLM — OpenAI GPT-4o-mini with Gemini fallback
  • Vector search — Qdrant (optional; keyword fallback when unavailable)
  • PDF reports — fpdf2 with NBFC credit assessment layout
  • Deploy — Docker Compose (Postgres + Qdrant + backend + frontend)

Flexible Parsing Engine

The core innovation for Indian banks is column_map.py — a shared module used by both PDF and Excel parsers. Instead of hardcoding “row 0 is the header”, it:

  1. Scans the first 20 rows of any table or spreadsheet
  2. Scores each row by how many standard columns it matches (date, narration, debit, credit, balance)
  3. Picks the highest-scoring row as the header
  4. Maps column synonyms — e.g. “Withdrawal Amt.” → debit, “Deposit Amt.” → credit
  5. Converts each data row into a normalized ParsedTransaction

Bank detection uses regex signatures on PDF text — no ML required for the initial routing step. Unknown banks fall through to a generic table parser that still uses flexible header detection.

Bulk Upload

NBFC agents often receive 3–6 months of statements from different accounts. The API exposes POST /api/statements/upload/bulk accepting multiple files in one request. The frontend supports drag-and-drop multi-select and redirects to the Chat tab with all processed statements selected for combined analysis.

LangGraph AI Analysis Pipeline

When a user clicks Run AI Analysis, a LangGraph state machine runs these agents in sequence:

  • Classify — LLM batch categorization with rule-based fallback
  • Fraud Agent — circular transactions, cash-heavy accounts, suspicious salary patterns, balance spikes
  • Cashflow Agent — monthly net flow, negative balance months, health score
  • EMI Agent — recurring fixed debits, total EMI, burden as % of credits
  • Salary Agent — monthly salary credits, income stability score (variance-based)
  • Risk Summary — consolidated narrative for underwriters

Each agent returns structured JSON stored in statement_insights. The dashboard displays risk level, income stability, cashflow health, and individual risk flags with severity tags.

AI Chat Assistant (RAG + Hinglish)

Loan agents in India often prefer simple Hindi-English questions: “EMI kitna hai?”, “Bounce hua kya?”, “Loan mil sakta hai?”. The chat layer supports:

  • Single statement — deep dive into one customer file
  • Multiple statements — combined analytics across accounts/months
  • LLM answers — OpenAI/Gemini with layman-friendly system prompt
  • Rule-based fallback — works even when API quota is exhausted
  • Vector search — Qdrant indexes transaction descriptions for relevant context

The RAG service builds an analytics context block (totals, salary, EMI, bounces, categories) and attaches matching transactions before calling the LLM — so answers stay grounded in actual statement data.

NBFC Credit Assessment PDF Report

The downloadable report follows a standard underwriter format — not a plain text dump. Sections include:

  • Account information with masked account number
  • Executive summary with color-coded risk banner
  • Financial overview KPI cards
  • Category-wise spending breakdown table
  • Income analysis and salary stability score
  • EMI obligations and burden assessment
  • Cashflow health and savings rate
  • Risk flags table (severity, type, observation)
  • Monthly transaction trend table
  • Credit scoring summary with underwriter recommendation
  • Legal disclaimer footer on every page

API Design

RESTful endpoints exposed via FastAPI with auto-generated OpenAPI docs at /docs:

GET  /health                          — health + config status
GET  /api/statements                  — list all statements
POST /api/statements/upload           — single file upload
POST /api/statements/upload/bulk      — multiple files
GET  /api/statements/{id}             — statement + transactions
GET  /api/statements/{id}/analytics   — financial analytics
POST /api/statements/{id}/analyze     — run LangGraph pipeline
GET  /api/statements/{id}/insights    — AI insights JSON
GET  /api/statements/{id}/report      — download PDF report
POST /api/chat                        — multi-statement chat

Cursor AI Development Agent

The repo includes an AGENTS.md file and Cursor skills/rules so future development stays consistent — bank parser conventions, API patterns, and phase-wise roadmap are documented for AI-assisted coding. This reduced iteration time when adding bulk upload, flexible parsers, and the chat tab.

Comparison: This Build vs Production GPU System

I also lead the production bank statement platform at Airtel Payments Bank using Ollama on NVIDIA GPU. The two systems complement each other:

  • Production (Django + Ollama + Ray) — high volume, on-premises LLM, Celery queues, GPU batch inference
  • This build (FastAPI + LangGraph + Next.js) — modern MVP, cloud LLM APIs, agent orchestration, NBFC UX focus

Lessons from production — idempotent processing, hybrid rule+LLM parsing, confidence logging — directly informed this FastAPI + LangGraph architecture.

Results & Key Lessons

  1. Flexible headers beat bank-specific parsers — one synonym-based mapper handles more formats than 13 hardcoded parsers
  2. Always ship a rule-based fallback — LLM APIs hit quota limits; the assistant still answers from analytics
  3. LangGraph keeps agents testable — each agent (fraud, EMI, salary) runs independently in the pipeline graph
  4. Reports must match underwriter expectations — structured sections, risk banners, and recommendations — not raw JSON
  5. Hinglish chat increases adoption — loan agents ask questions naturally; the UI should match
