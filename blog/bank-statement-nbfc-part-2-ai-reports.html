NBFC Bank Statement Platform — Part 2: LangGraph AI, Hinglish Chat & Credit Reports

By Nitesh Kumar Singh · NKSCoder · June 19, 2026 · ← Part 1

Part 2 of 2. Once transactions are extracted and categorized (see Part 1), the intelligence layer kicks in: a LangGraph multi-agent pipeline for risk analysis, a Hinglish chat assistant for loan agents, and downloadable NBFC-standard credit assessment PDF reports.

From Data to Decisions

Raw transactions answer “what happened.” Underwriters need “what does it mean for this loan file?” The AI layer transforms aggregated analytics into:

  • Risk score and severity-tagged flags
  • Income stability assessment
  • EMI burden relative to credits
  • Cashflow health over the statement period
  • Natural-language Q&A for field agents
  • Printable credit memorandum for file documentation

LangGraph Agent Pipeline

Clicking Run AI Analysis triggers a compiled LangGraph state machine. Each node reads and writes to a shared PipelineState dict: 

extract → classify → fraud_check → cashflow_check
                              → emi_check → salary_check → summarize_risk → END

1. Classify Node

Attempts LLM batch categorization via GPT-4o-mini. If API unavailable, falls back to rule-based categorizer. Updates transaction category fields in PostgreSQL when LLM returns confident labels.

2. Fraud Agent

Detects patterns that raise underwriting concern:

  • Circular transactions — same amount repeated 6+ times (possible round-tripping)
  • Cash-heavy account — ATM/cash debits > 40% of total outflow
  • Suspicious salary — round-number credits without salary keywords
  • Balance spikes — sudden large credits before loan application window

Outputs: risk_level (low/medium/high), risk_score 0–100, flags[] with severity.

3. Cashflow Agent

  • Groups transactions by month; computes net flow per month
  • Counts months with negative net flow
  • Assigns cashflow_health: good / moderate / poor
  • Tracks average balance when available

4. EMI Agent

  • Identifies EMI and loan repayment debits by category
  • Detects recurring fixed debits (same amount 2+ times) as likely EMIs
  • Calculates total_emi and emi_burden_pct = EMI / total credits

5. Salary Agent

  • Finds salary-category credits; detects large monthly credits as proxy
  • Computes avg_monthly_salary and salary_months count
  • income_stability_score — 100 minus coefficient of variation across salary credits

6. Risk Summary Node

Merges all agent outputs into a single underwriter narrative stored as risk_summary — e.g. “Overall risk: medium (55/100). Average monthly income ~INR 75,000. EMI burden: 21% of credits.”

Dashboard Presentation

After analysis, the dashboard shows:

  • Risk level badge with score
  • Income stability and cashflow health cards
  • Individual risk flags with severity tags
  • Updated category breakdown on transactions table
  • Download PDF Report button when generation succeeds

Hinglish Chat Assistant

Loan agents in India don't write formal queries. They ask: “EMI kitna hai?”, “Bounce hua kya?”, “Loan mil sakta hai?”, “Summary batao”. The chat layer is built for this.

Single vs Multiple Statements

  • Single mode — one statement selected; deep analytics context
  • Multiple mode — checkbox select; combined totals across accounts/months
  • Bulk upload auto-redirects to Chat with all statement IDs pre-selected

RAG Architecture

  1. Build analytics context block (credits, debits, salary, EMI, bounces, categories)
  2. Search Qdrant vector store for relevant transaction descriptions
  3. Keyword fallback if Qdrant unavailable
  4. Send context + user question to GPT-4o-mini (Gemini fallback)
  5. Rule-based answer if LLM quota exhausted — never returns blank

System prompt enforces layman language: “money coming in” not “credits”, lakh/crore formatting, Hinglish when user writes in Hindi.

Example Rule-Based Answers (No LLM)

  • “Salary kitni hai?” → avg salary from analytics + stability score
  • “Bounce hua kya?” → bounce count with lender warning if > 0
  • “Loan mil sakta hai?” → rough eligibility estimate from disposable income
  • “Summary batao” → full period overview in simple language

NBFC Credit Assessment PDF Report

Underwriters need a document for the loan file — not a JSON dump. The report agent generates a multi-section PDF with fpdf2:

  1. Header bar — report title, reference ID, confidential tag
  2. Account information — bank, masked account number, analysis period
  3. Executive summary — color-coded risk banner (green/amber/red)
  4. Financial overview — KPI row: credits, debits, net flow, avg balance
  5. Category breakdown table — spend % by category
  6. Income analysis — salary, stability score, pattern assessment
  7. Obligations & EMI — total EMI, burden %, bounce count
  8. Cashflow analysis — health, savings rate
  9. Risk flags table — severity, type, observation
  10. Monthly trend table — month-wise credit/debit/net
  11. Credit scoring summary — parameter scores with remarks
  12. Underwriter recommendation — proceed / enhanced DD / manual review
  13. Disclaimer footer on every page

Unicode rupee symbols are sanitized to “INR” for PDF font compatibility.

API Endpoints (Intelligence Layer)

POST /api/statements/{id}/analyze   — run LangGraph pipeline
GET  /api/statements/{id}/insights    — fetch AI insights JSON
GET  /api/statements/{id}/report      — download PDF report
POST /api/chat                        — { statement_ids[], question }
GET  /health                          — API status + masked key config

Comparison: This Build vs Production GPU System

I also lead production bank statement analysis at Airtel Payments Bank (Ollama + NVIDIA GPU):

  • Production — Django, Celery, Ray, on-premises Ollama, thousands of statements/day
  • This NBFC build — FastAPI, LangGraph, Next.js, cloud LLM APIs, agent UX focus

Shared lessons: hybrid rule+LLM parsing, idempotent processing, always ship non-AI fallbacks.

Key Lessons

  1. LangGraph isolates agents — test fraud logic without running full pipeline
  2. Never depend only on LLM APIs — quota limits are real; rule fallback is mandatory
  3. Chat in the user's language — Hinglish adoption is higher than formal English UI
  4. Reports = trust — structured PDFs signal professionalism to NBFC partners
  5. Multi-statement chat — agents often have 3–6 months across accounts; support it from day one

Roadmap

  • OCR for scanned/image PDFs
  • Loan eligibility scoring engine (Phase 3)
  • Ollama local LLM — same stack as production GPU system
  • Multi-tenant SaaS with NBFC API keys
  • CIBIL integration and white-label reports
