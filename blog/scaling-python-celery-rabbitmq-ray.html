Scaling Python Workers with Celery, RabbitMQ, and Ray

By Nitesh Kumar Singh · June 8, 2026

When a Django API starts receiving thousands of bank statements daily, synchronous request handling breaks immediately. Here's how I design Python worker pipelines using Celery, RabbitMQ, and Ray for production financial workloads.

Why Three Tools?

Each solves a different scaling problem:

  • Celery — reliable async task execution with retries, scheduling, and result backends
  • RabbitMQ — durable message broker with priority queues and dead-letter exchanges
  • Ray — distributed compute for CPU-heavy Pandas operations across multiple cores/machines

Celery handles orchestration. RabbitMQ ensures no task is lost during deploys. Ray parallelizes the heavy data crunching that would block a single worker for minutes.

Queue Design

I separate queues by priority and resource profile:

  • ingest.high — small files, user-facing latency sensitive
  • ingest.bulk — batch uploads, overnight processing
  • llm.gpu — GPU-bound Ollama inference tasks (concurrency limited)
  • notify — webhooks and email, isolated from ingest

Dead-letter queues capture failed tasks after max retries. Every DLQ message triggers an alert — silent failures are unacceptable in banking pipelines.

Idempotency Is Non-Negotiable

Statements get re-uploaded. Workers restart mid-task. RabbitMQ delivers at-least-once. Every Celery task must be idempotent: use a deterministic job ID derived from file hash + version, upsert instead of insert, and store processing state in PostgreSQL before acknowledging the message.

Ray for Data Parallelism

A 50MB CSV with 200K rows cannot be processed in one Pandas call on a single core. Ray lets me partition the dataframe, distribute chunks to workers, and aggregate results — often 4–8x faster on the same hardware. Ray tasks are stateless; Celery triggers the Ray job and polls for completion.

Monitoring & Operations

  • Flower for Celery worker visibility
  • RabbitMQ management UI for queue depth alerts
  • Structured JSON logging with correlation IDs across services
  • Graceful worker shutdown — finish current task before SIGTERM timeout
