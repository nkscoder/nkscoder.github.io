NBFC Bank Statement Platform — Part 1: Architecture, Upload & Multi-Bank Parsing

The NBFC Problem

Every day, loan agents and NBFC underwriters receive bank statements as PDF downloads or Excel exports. Before any loan decision, someone must manually answer:

  • Is salary coming regularly? How much per month?
  • What EMIs or loan payments are already running?
  • Any cheque bounces or return transactions?
  • What is the net cash flow and spending pattern?
  • Can this customer afford another EMI?

Doing this by hand for 50–200 transactions per statement takes 30–60 minutes. Multiply by multiple accounts and months of data, and the process becomes a bottleneck for loan origination teams.

Platform Goals

I designed the platform around five core workflows:

  1. Ingest — accept PDF, Excel (.xlsx/.xls), and CSV in single or bulk upload
  2. Extract — normalize transactions regardless of bank export format
  3. Categorize — tag Salary, EMI, UPI, Shopping, Bounce, Transfer automatically
  4. Aggregate — compute monthly totals, category breakdowns, salary averages
  5. Present — dashboard analytics ready for underwriter review (Part 2 adds AI layer)

Tech Stack (Part 1 Layer)

  • FastAPI — async upload endpoints, Pydantic validation, OpenAPI at /docs
  • SQLAlchemy 2 — ORM models for statements, transactions, insights
  • PostgreSQL — relational storage with audit-friendly status fields
  • pdfplumber — PDF table extraction with text-line fallback
  • openpyxl — Excel reading with data_only=True for formula cells
  • Next.js 15 — upload drag-and-drop, statement list, analytics tables
  • Docker Compose — Postgres + backend + frontend for local and server deploy

Database Design

Three core tables power the platform:

statements

  • id, filename, file_path
  • bank_name, account_number — detected during parse
  • status — processing → processed or failed
  • error_message — user-visible reason when extraction fails

transactions

  • txn_date, description, debit, credit, balance
  • category — assigned by categorizer (Salary, EMI, UPI, etc.)

statement_insights

  • insights_json — AI pipeline output (populated in Part 2)
  • report_path — generated PDF location

Upload Flow: Single & Bulk

Single upload uses POST /api/statements/upload. The file is saved to a secure upload directory with a UUID prefix, a statement row is created with status=processing, and process_statement() runs synchronously before the API responds.

Bulk upload uses POST /api/statements/upload/bulk with multiple files. Each file is processed independently — partial success is supported. The frontend drag-and-drop zone accepts multi-select; on success with 2+ statements, users land on the Chat tab with all IDs selected.

Supported formats: .pdf, .xlsx, .xls, .csv

The Parsing Challenge: No Fixed Headers

Indian bank exports are inconsistent. Real examples I've handled:

  • HDFC — title rows before headers; columns named “Withdrawal Amt.” / “Deposit Amt.”
  • SBI — “Txn Date”, “Value Date”, “Description” in different positions
  • ICICI — Excel exports with account summary rows above transaction table
  • Axis / Kotak — PDF tables that break across pages mid-row

Hardcoding “header is always row 0” fails on the second file. The solution is header row scoring — not bank-specific row numbers.

column_map.py — Flexible Column Detection

The shared parser module works for both PDF tables and Excel sheets:

  1. Scan first 20 rows for the best header candidate
  2. Score each row by matched columns: date, description, debit, credit, balance
  3. Map synonyms — “Narration”, “Particulars”, “Transaction Details” → description
  4. Parse amounts — strip Dr/Cr, handle (1,234.56) negatives
  5. Emit normalized ParsedTransaction objects
  6. Deduplicate — hash on date + description + amounts to avoid double-counting

Supported banks via signature detection: HDFC, SBI, ICICI, Axis, Kotak, PNB, BOB, Canara, Union, IDBI, Yes, IndusInd, Bandhan — plus generic fallback.

PDF vs Excel Parsing Paths

PDF (pdfplumber)

  • Extract tables per page with flexible header detection
  • If table extraction fails, split text lines on whitespace gaps
  • Regex-detect amounts ([\d,]+\.\d{2}) and dates (DD/MM/YYYY, DD-MMM-YYYY)
  • Read first 2 pages for bank name and account number patterns

Excel / CSV (openpyxl)

  • Load all rows; run same find_header_row() logic
  • Try UTF-8, UTF-8-BOM, and Latin-1 encodings for CSV
  • Positional fallback if no header match — assume date, description, amounts columns

Transaction Categorization

Before AI analysis, a rule-based categorizer tags transactions using description keywords:

  • Salary — “SALARY”, “NEFT CR-SALARY”, corporate credit patterns
  • EMI — “EMI”, “LOAN”, “HDFC LTD”, “BAJAJ FIN” debit patterns
  • UPI — “UPI/”, “@paytm”, “@ybl”
  • Bounce — “CHQ RET”, “INSUFF”, “BOUNCE”
  • ATM — “ATM WDL”, “CASH WDL”

LLM batch categorization runs during the AI pipeline (Part 2) when OpenAI key is configured.

Analytics Engine

compute_analytics() aggregates processed transactions into:

  • Total credit, debit, net flow, transaction count
  • Monthly summary — credit/debit per YYYY-MM
  • Category summary — spend and income by category
  • Avg salary credit, total EMI, EMI burden %, bounce count
  • Average monthly balance (when balance column present)

These metrics feed the dashboard immediately after upload — no AI required.

Failure Handling

When zero transactions are extracted, the statement is marked failed with a clear message: “No transactions extracted. Try Excel export or a different PDF format.” This prevents empty dashboards and guides agents to re-upload a better export.

Deployment

Docker Compose runs four services:

  • postgres — port 5432
  • qdrant — vector DB for chat (Part 2)
  • backend — uvicorn on port 8000
  • frontend — Next.js on port 3000

Environment variables: DATABASE_URL, OPENAI_API_KEY, GEMINI_API_KEY, QDRANT_URL
